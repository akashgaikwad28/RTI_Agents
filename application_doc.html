01

High Level System Flow

The RTI-Agent operates as an asynchronous, state-aware multi-agent system. It transitions from a stateless request-response model into a persistent, checkpointed workflow.

End-to-End Lifecycle

  1. Ingress — User submits a query via FastAPI.
  2. Hardening — Middleware performs Auth, Rate Limiting, Sanitization, and PII Masking.
  3. Orchestration — LangGraph initializes an RTIAgentState and begins node execution.
  4. Reasoning — Agents (Router, Formatter, Classifier, Reviewer) collaborate using multi-model orchestration (Groq/Gemini).
  5. Grounding — Retrieval Agent pulls facts from FAISS and Government Portals.
  6. Governance — Graph interrupts at approval_node for Human-in-the-Loop review.
  7. Persistence — TrackerNode finalizes records in MongoDB and sends notifications.

02

Complete Request Flow

Step-by-step execution of POST /api/v1/submit:

  1. Request Validation — FastAPI validates the RTISubmitRequest schema (Pydantic).
  2. Request ID Generation — RequestIDMiddleware generates a UUID and injects it into X-Request-ID headers and request.state.
  3. Sanitization — security/sanitizer.py strips HTML, SQL injection, and 15+ prompt injection patterns.
  4. Graph Initialization — api/routers/rti.py creates the initial RTIAgentState: request_id and thread_id set, raw_query preserved, status set to "pending".
  5. Execution Start — graph.ainvoke(initial_state, config) is called.
  6. Checkpointing — LangGraph immediately saves the initial state into rti_checkpoints.db indexed by thread_id.
  7. Async Boundary — The graph runs asynchronously. Since interrupt_before=["approval_node"] is set, it runs all nodes until the Approval phase.
  8. API Response — Endpoint returns a 202 Accepted with request_id and stream_url.

05

RAG Pipeline Flow

The RAG pipeline ensures the system is grounded in actual RTI law and government structure.

RAG Lifecycle

  1. Ingestion — rag/ingestion/pipeline.py loads PDFs/JSONs.
  2. Chunking — RecursiveCharacterTextSplitter creates 512-token chunks.
  3. Embedding — GeminiEmbedder generates 768-dim vectors.
  4. Indexing — FAISS stores vectors locally in data/vector_store.
  5. Retrieval — RetrievalNode calls retriever.py.
  6. Caching — Redis stores SHA256(query+dept) → Context mapping.
RAG Sequence — Cache-First Retrieval

Memory Architecture

The system uses a tiered memory model to handle the long-running RTI lifecycle.

Tier 1
Short-Term
Ephemeral RTIAgentState TypedDict holding in-flight data.
Tier 2
State Persistence
SQLite Checkpointer saves state at every node transition. Allows resume after crash or HITL pause.
Tier 3
Semantic
FAISS index providing long-term contextual facts.
Tier 4
Episodic
MongoDB rti_requests storing completed applications.
Tier 5
Thread
MongoDB conversation_threads linking multiple requests to one user.
07

HITL (Human-in-the-Loop) Flow

This is the most critical flow for governance.

  1. Interrupt — Graph reaches approval_node. Because interrupt_before is set, execution halts.
  2. Notification — approval_node saves a "pending" status to Mongo and sends an email.
  3. Wait — The graph process exits. State is safely persisted in SQLite.
  4. Action — User calls POST /api/v1/approve/{request_id}.
  5. Resume — FastAPI calls graph.ainvoke(resume_state, config). LangGraph finds the checkpoint, updates state with approval_status="approved", and execution resumes exactly where it left off.

Observability & Security

Observability

  1. Prometheus — Emits metrics at each node end (rti_agent_duration_seconds).
  2. LangSmith — Captures full graph trace, including inputs/outputs of every LLM call.
  3. JSON Logs — Every log includes request_id and agent name for correlation in ELK/Grafana.

Final Engineering Review

▲ Architecture Strengths

  • ●
    Resilience — Checkpointing ensures no request is lost mid-flight.
  • ●
    Governance — HITL is baked into the graph topology, not an afterthought.
  • ●
    Hybrid RAG — Combining local FAISS with live Tool-Calling ensures up-to-date context.

⚠ Bottlenecks & Risks

  • ●
    Async Risk — Full async requires careful connection pool management under high load.
  • ●
    Graph Complexity — As more agents are added, a single TypedDict state can become bloated.
  • ●
    Token Cost — Reflection loops can double token usage for failing queries.
